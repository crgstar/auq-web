ライブラリを Python で固める方針で OK ?

single

auq-web の起動部分は 「呼び出し側に何もインストールさせない」 のが理想です。Skill が /auq で呼ばれた瞬間に、ユーザーの環境で確実に動かす必要があり、事前準備が要る言語を選ぶと初回利用で「あれ、動かない」と詰まる人が出て、Skill の "気軽に呼べる" 性質を失います。

macOS には Python 3 が pre-installed されています。brew install も npm install も要らず、http.server・argparse・json がすべて標準ライブラリにあります。1 ファイルで HTTP サーバ + JSON 入出力 + 引数パーサを完結できる、数少ない言語の一つです。

Bun は速度面で魅力的で、Web API が網羅されているので 20 行程度で書けます。ただし brew install bun が前提になり、Skill を 誰かに配る 想定ではハードルが上がります。Node は標準では何もないので追加 npm が必要、しかも Mac には pre-installed されていません。

言語事前準備1-shot 行数目安標準ライブラリ範囲
Python不要 (macOS 標準)~50HTTP, JSON, argparse 揃う
Bunbrew install bun~20Web API 網羅
Node場合により要~70http モジュールのみ。npm 必要な機能多め

「Skill は気軽に呼べる」「初回ユーザに事前準備を強いない」を最優先するなら、Python が最も安全な選択。書く行数が多くても、それは一度書けば済む話。呼ばれる側のコスト ではなく 書く側のコスト です。

ただし Mac 以外の環境 (CI / Linux 開発機 / Codespaces) で動かしたいなら状況が変わります。Linux も Python 3 が標準で入っている事が多いですが、最近は python3 が入っていないディストリも稀にあります。汎用性で見ても Python に分。

MVP に入れる機能を選んで（複数可）

multi

MVP に何を入れるかで、初回リリース後の 方向性 が決まります。多すぎると複雑化して保守が重く、少なすぎると "使えない" と判断されて誰も呼ばなくなる。最低限「呼んで意味がある」状態の境界線を引きたい所です。

このスキルの真価は Claude が自由に表現された質問を投げられる事。これは骨格として常に提供されます。一方で AskUserQuestion 互換 (1 ページに最大 4 問)・multiSelect・タイムアウト・port-conflict 画面は、それぞれ独立した拡張で、入れるかは判断要です。

特に port-conflict 画面は実装時の落とし穴になりやすく、「7777 既に埋まってる時に何が起きるか分からない」を最初から潰しておく価値はあります。逆に multiSelect や複数質問は AskUserQuestion 互換性のためで、互換を諦めるなら後回しにできます。

判断基準: 「ない時に困るのは誰か」。エラー画面は誰でも困る。multiSelect が無いと困るのは "互換性が必要な呼び出し側"。それぞれの "困る側" の優先度で線引きしてください。

Q4 / 5

タイムアウト時の挙動はどう返したい？

single

このスキルは Bash 同期ブロックではなく Monitor 経由の背面受信。サーバは background で起動し続けるため、ユーザーが応答しないケース (画面を閉じた・席を立った) で永遠に生かしておくわけにはいきません。タイムアウトの仕様はこのスキルの "終わり方" を決める重要な決定です。

そのため Skill は起動時に timeoutSec を受け取り、サーバ側でカウントダウン。指定秒を過ぎても POST が来なければサーバを終了し、Claude には JSON で通知が飛びます。問題は その通知の形と、Claude の続きの動き。

典型的には次の 2 形式が考えられます:

# A. error 扱い (exit 1 + stderr)
{ "event": "timeout", "elapsedSec": 300 }

# B. answer JSON に flag (exit 0)
{ "event": "answer", "timedOut": true, "answers": {}, "elapsedSec": 300 }

A は Bash の exit code でも区別できる代わりに、Claude 側で「これは timeout だ、retry すべきかユーザに確認すべきか」を毎回判断する必要があり、呼び出し側の負担が大きく なります。例外的な制御フロー (try/catch) で扱う事になり、コードが分岐がちです。

一方 B は構造化レスポンスとして一貫性があり、Claude は timedOut の真偽だけ見れば済みます。if (result.timedOut) 一行で分岐できる。「明示的に答えなかった」と「答えたが timeout 直前に submit した」が紛れる懸念はあるため、elapsedSec 等のメタ情報も同梱したい所。さらに partialAnswers (timeout 直前に答えていた質問) を入れれば部分回答も拾える設計になります。

スキルの第一目標は エンジニアにとっての扱いやすさ。呼び出し側の分岐が浅くなる方を選びたい。if (result.timedOut) で済むのと、try { ... } catch (TimeoutError) が必要なのでは、可読性とエラー処理の負担が違います。「失敗を例外で表すか結果値で表すか」の Go 的選択でもあります。

Q5 / 5

サーバ起動方式はどちらにする？

single

Skill が Bash で起動するサーバを foreground で 1-shot 動作させるか、background で常駐型にするかで全体の動きが変わります。両方とも実装可能ですが、Claude 側の体験が大きく異なるため、最初に決めておきたい設計判断です。

foreground 1-shot は 同期 RPC です。Bash が server.py を呼ぶ → ブラウザでユーザが answer → server がそれを stdout で返して exit → Bash の戻り値として Claude に返る。シンプルで、デバッグもしやすい。ただし Claude は応答が来るまで待機状態になり、他の事ができません。

background + Monitor は 非同期 push。Bash で server を background 起動し、Monitor ツールが server の stdout を tail。ユーザが answer したタイミングで Monitor が Claude に通知 → Claude が次のアクションを開始。Claude は server 起動後、別のタスクと並行作業できます。

判断基準: 「Claude の応答時間」を最優先するなら B。たとえば 「ユーザに 1 問質問しつつ、別ファイルを読み始める」 という並行動作は B でしか成立しません。逆に「実装を最小に保ちたい」なら A も合理的。 A→B の移行は subprocess.Popen の起動方式変更だけで済むので、後からでも切り替えられます。

kumazaki さんの指定では B (Monitor 背面受信)。Skill の使われ方が「Claude が長時間考える間に質問を挟む」ケースを想定していて、Claude が固まらない事を優先しています。最初から B で書く方が型が固まる。
